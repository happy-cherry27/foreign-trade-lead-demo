解决的不是“发邮件”，而是销售线索处理效率。

外贸询盘邮件里常常混杂客户信息、采购意图、预算、数量和交期。系统要做的是把这些信息变成销售可审核、可排序、可同步的业务对象。

PAIN 01

人工录入慢

销售需要从邮件中复制客户姓名、公司、产品、数量、预算等字段，重复劳动多，容易漏填。

PAIN 02

高意向客户容易被延迟

urgent、deadline、within 3 days 等信号如果没有被及时识别，可能错过报价和成交窗口。

PAIN 03

AI 结果需要可审核

外贸客户关系敏感，AI 不能直接写入 CRM。系统必须保留人工审核和日志记录。

从多渠道询盘进入，到飞书同步，完整闭环。

这条链路可以通过手动录入、批量导入、n8n webhook 或微信聊天文本触发。确认后的线索才能进入飞书或正式 CRM。

01询盘进入邮件粘贴、批量导入、n8n 调用 `/api/webhooks/email`，或微信聊天文本导入。
02字段抽取抽取客户、公司、国家、产品、预算、数量、电话等字段。
03线索评分根据紧急程度、商业价值、联系完整度等计算 lead_score。
04跟进建议生成英文回复草稿和缺字段追问，帮助销售快速响应。
05人工审核进入 pending_review，销售确认、修改或拒绝。
06同步留痕confirmed 后同步飞书，成功或失败都写入 timeline。

技术架构保持轻量，但边界清楚。

后端使用 FastAPI + SQLite，前端使用原生 HTML/CSS/JavaScript。模块已经拆分，便于说明职责边界。

Core API
POST /api/webhooks/email
POST /api/leads/extract
POST /api/leads/extract-wechat
POST /api/leads/import-batch
PATCH /api/leads/{id}/review
POST /api/leads/{id}/sync/feishu
GET  /api/leads/{id}/logs

backend/main.py

FastAPI 路由入口，负责请求编排。

backend/extractor.py

邮件/微信聊天抽取、字段级证据和基础意图识别。

backend/scoring.py

lead_score 与 score_breakdown 评分规则。

backend/followup.py

英文回复草稿和缺字段追问生成。

backend/integrations/feishu.py

飞书 Bitable OpenAPI 同步适配。

frontend/app.js

抽取、审核、批量导入、同步、timeline 交互。
