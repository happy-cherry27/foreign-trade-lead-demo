# 2026-06-16 FastAPI 闭环学习记录

## 学习背景

我之前学 FastAPI 已经是将近一个半月前，很多基础概念忘记了。今天看到 `backend/main.py` 时，一开始不太能看懂，因为这份代码不是单独的 FastAPI 教程，而是把后端接口、SQLite 数据库、邮件抽取逻辑、前端页面返回、人工审核和日志记录都放在了一个工程里。

这次学习目标不是一次性看懂每一行代码，而是先建立项目运行链路的整体认知。

## 今天理解到的核心链路

完整 demo 链路是：

```text
/health
确认后端服务是否活着

/
返回前端页面

POST /api/leads/extract
接收邮件正文，抽取客户线索

POST /api/leads
保存抽取结果到 SQLite 数据库

GET /api/leads
从数据库读取线索列表

GET /api/leads/{id}
查看某条线索详情

PATCH /api/leads/{id}/review
人工确认或拒绝线索

GET /api/leads/{id}/logs
查看 AI 抽取日志和审核记录
```

## 1. `/health` 的作用

`/health` 是健康检查接口。

它不处理业务，不抽取邮件，不查数据库，不保存线索。它只回答一个问题：

后端服务现在是否活着？

访问：

```text
http://127.0.0.1:8000/health
```

看到：

```json
{"status":"ok"}
```

说明后端服务能正常响应请求。

我今天形成的理解：

```text
/health = 问后端：“你还活着吗？”
后端回答：“我活着。”
```

调试顺序也因此变清楚：

1. 先看 `/health`，判断后端有没有启动。
2. 再看 `/`，判断前端页面能不能加载。
3. 再测业务接口，判断抽取、保存、审核这些功能有没有问题。

## 2. `/` 的作用

`/` 是首页接口。

它对应代码里的逻辑是：

```python
@app.get("/")
def index():
    return FileResponse(FRONTEND_DIR / "index.html")
```

我的理解：

当浏览器访问 `http://127.0.0.1:8000/` 时，后端把 `frontend/index.html` 这个前端页面返回给浏览器。

所以：

- `/health` 证明后端活着
- `/` 证明后端能把前端页面发给浏览器

## 3. `POST /api/leads/extract` 的作用

这个接口负责“抽取客户线索”。

页面上点击“填入示例”再点击“抽取客户线索”后，前端会把邮件正文发给后端。

后端收到邮件后，调用 `extract_lead()`，从邮件里抽取：

- 客户姓名
- 邮箱
- 公司
- 国家
- 电话
- 产品需求
- 预算
- 数量
- 紧急程度
- 优先级
- 跟进建议
- 判断依据
- 置信度

我今天跑通后理解到：

```text
浏览器页面不是自己在变魔术，
它是在调用后端的 POST /api/leads/extract 接口。
```

面试里可以这样说：

“我先做了一个邮件抽取接口，前端把客户邮件正文通过 `POST /api/leads/extract` 发给后端，后端调用抽取逻辑，把客户信息、产品需求、优先级、跟进建议和判断依据返回给前端展示。”

## 4. `POST /api/leads` 的作用

这个接口负责“保存线索”。

点击“保存线索”后，前端会把刚才抽取出来的客户信息发送给后端。

后端会做两件事：

1. 把客户线索写进 SQLite 数据库里的 `leads` 表。
2. 把 AI 抽取过程写进 `ai_extraction_logs` 表。

这一步的意义是：

页面不只是展示了一下抽取结果，而是把数据真正存下来了。

我的理解：

```text
POST /api/leads/extract = 只抽取，不保存
POST /api/leads = 把抽取结果正式存进数据库
GET /api/leads = 从数据库读取所有线索，展示到列表
```

## 5. `GET /api/leads` 和 `GET /api/leads/{id}` 的作用

`GET /api/leads` 用来读取线索列表。

保存线索后，页面左下方的线索列表会出现记录，这说明前端通过 `GET /api/leads` 从数据库拿到了数据。

`GET /api/leads/{id}` 用来读取某一条线索详情。

当我点击列表中的某条线索，右侧详情区出现字段、原始邮件和审核按钮，这说明详情接口跑通了。

我的理解：

```text
GET /api/leads = 看所有线索
GET /api/leads/{id} = 看某一条线索
```

## 6. `PATCH /api/leads/{id}/review` 的作用

这个接口负责人工审核。

当我点击“确认线索”或“拒绝线索”时，前端调用这个接口，把线索状态从：

```text
pending_review
```

改成：

```text
confirmed
```

或：

```text
rejected
```

这一步是整个 demo 的业务关键点。

因为外贸客户关系比较敏感，AI 不应该直接替销售发消息，也不应该直接把所有判断当成最终事实。AI 先给建议，人再审核，这样更符合真实业务风险控制。

## 7. `GET /api/leads/{id}/logs` 的作用

这个接口用来查看日志。

日志包括两类：

- AI 抽取日志：AI/规则当时抽出了什么、置信度是多少、判断依据是什么
- 人工审核记录：人后来确认或拒绝了什么、是否修改字段、备注是什么

我的理解：

日志的作用不是为了好看，而是为了可追溯。

如果面试官问“AI 抽错怎么办”，可以回答：

“我没有让 AI 结果直接成为最终客户资料，而是保存了抽取日志和人工审核记录。这样后续可以复盘 AI 为什么这么判断，也能看到人工修改了哪些字段。”

## 今天真正掌握的内容

我今天不需要假装自己完全看懂了所有代码。

我真正掌握的是项目的运行链路：

```text
服务活着 -> 页面加载 -> 邮件抽取 -> 保存数据库 -> 列表展示 -> 查看详情 -> 人工审核 -> 日志追踪
```

这已经比“完全不知道项目怎么跑”前进了一大步。

## 下一步学习目标

下一步不急着加功能，先继续理解 4 个 FastAPI 基础概念：

1. `app = FastAPI()`：创建后端应用。
2. `@app.get/post/patch(...)`：定义接口。
3. `BaseModel`：规定前端传来的数据长什么样。
4. `sqlite3.connect()`：连接 SQLite 数据库。

## 面试可用表达

“我这次先没有追求把所有功能都做复杂，而是先把一个工程闭环跑通。我从 `/health` 开始确认后端服务正常，再通过 `/` 返回前端页面。用户在页面里粘贴客户邮件后，前端调用 `POST /api/leads/extract`，后端抽取客户信息、需求、优先级和跟进建议。确认无误后，再通过 `POST /api/leads` 保存到 SQLite。线索列表用 `GET /api/leads` 展示，详情页用 `GET /api/leads/{id}` 查看，最后人工审核通过 `PATCH /api/leads/{id}/review` 更新状态，并通过日志接口保留 AI 抽取和审核记录。”

## 反思

今天最大的收获不是“我已经完全掌握 FastAPI”，而是我知道了如何重新进入一个工程：

1. 先确认服务活着。
2. 再确认页面能加载。
3. 再按用户操作顺序理解接口。
4. 最后把每个接口和业务动作对应起来。

这比直接硬啃整份代码更有效。
