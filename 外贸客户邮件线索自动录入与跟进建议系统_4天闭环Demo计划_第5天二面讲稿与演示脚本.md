# 外贸客户邮件线索自动录入与跟进建议系统

## 4 天工程闭环 Demo 计划 + 第 5 天二面讲稿与演示脚本

### 项目定位

这是一个面向外贸销售场景的 AI 工程落地 demo。

核心目标不是做完整 CRM，而是在 4 天内证明：我能把外贸公司真实业务里的“客户邮件线索录入与跟进”问题，拆成前端、后端、数据库、AI 抽取、人工审核、日志记录的可运行闭环。

第 5 天专门整理二面讲稿和演示脚本，把 demo 变成能拿去面试展示的资产。

### 项目题目建议

外贸客户邮件线索自动录入与跟进建议系统

### 最高优先级约束

4 天内必须闭环，不追求大而全。

闭环定义：

客户邮件正文输入 -> AI 或规则抽取字段 -> SQLite 入库 -> 页面展示线索列表 -> 详情页人工审核 -> 保存 AI 判断日志 -> 生成跟进建议。

只要这条链路跑通，就已经能回应面试官最关心的问题：你能不能把 AI 用在真实业务流程里，而不是只会讲 Prompt、Dify 或“我学得很快”。

## 一、最终 Demo 必须长什么样

### 用户能看到的页面

1. 邮件录入页

- 粘贴客户邮件正文
- 点击“抽取客户线索”
- 显示抽取结果预览
- 提交入库

2. 线索列表页

- 显示客户姓名、邮箱、公司、国家、产品需求、优先级、审核状态
- 支持按状态筛选：待审核、已确认、已拒绝

3. 线索详情页

- 显示原始邮件
- 显示 AI 抽取字段
- 显示判断依据和置信度
- 支持人工修改字段
- 支持确认或拒绝
- 显示跟进建议
- 显示日志记录

### 后端必须支持的接口

- `POST /api/leads/extract`：输入邮件正文，返回结构化抽取结果
- `POST /api/leads`：保存客户线索
- `GET /api/leads`：获取线索列表
- `GET /api/leads/{id}`：获取线索详情
- `PATCH /api/leads/{id}/review`：人工审核线索
- `GET /api/leads/{id}/logs`：查看 AI 抽取日志

### 数据库至少包含 3 张表

1. `leads`

- id
- name
- email
- company
- country
- phone
- product_need
- budget
- urgency
- priority
- status
- original_email
- follow_up_suggestion
- created_at
- updated_at

2. `ai_extraction_logs`

- id
- lead_id
- raw_email
- extracted_json
- confidence
- evidence
- model_name
- created_at

3. `review_records`

- id
- lead_id
- action
- before_json
- after_json
- reviewer_note
- reviewed_at

### AI 抽取字段

必须抽取：

- 客户姓名
- 邮箱
- 公司
- 国家
- 电话
- 产品需求
- 预算
- 紧急程度
- 优先级
- 适合跟进时间
- 跟进建议
- 判断依据
- 置信度

字段缺失时统一写 `unknown`，不要编造。

## Day 2：AI 抽取 + 日志 + 审核接口

### 当天目标

把“邮件正文”变成“结构化客户线索”，并留下 AI 判断依据。

Day 2 结束时，系统必须能完成：

输入邮件 -> 抽取 JSON -> 保存线索 -> 保存 AI 日志。

### 必做任务

1. AI 抽取模块

- 写抽取 Prompt
- 要求模型输出严格 JSON
- 缺失字段写 `unknown`
- 字段必须带判断依据
- 生成跟进建议
- 生成优先级：高、中、低

2. Mock 兜底模式

必须支持 mock 模式。

原因：如果 API key、网络、模型输出不稳定，demo 不能当场瘫痪。

建议逻辑：

- 有 API key：调用真实模型
- 没有 API key：使用本地规则或 mock JSON

3. 抽取接口

- `POST /api/leads/extract`
- 输入：邮件正文
- 输出：结构化 JSON、置信度、判断依据、跟进建议

4. 日志保存

- 保存 raw_email
- 保存 extracted_json
- 保存 confidence
- 保存 evidence
- 保存 model_name

5. 审核接口

- `PATCH /api/leads/{id}/review`
- 支持确认：`confirmed`
- 支持拒绝：`rejected`
- 支持修改字段后确认
- 保存修改前后差异

### 当天不要做

- 不追求 Prompt 完美
- 不做 RAG
- 不做联网搜索
- 不做复杂客户评分模型

### Day 2 验收标准

- 3 封测试邮件都能抽取字段
- 至少 1 封缺字段邮件能显示 `unknown`
- 至少 1 封紧急邮件能判定高优先级
- AI 日志能保存
- 人工审核状态能更新

### Day 2 面试沉淀

当天结束后记住这句话：

“我没有让 AI 直接改数据库或直接发消息，而是让它先生成结构化建议，再进入人工审核流程。这样更符合外贸客户关系比较敏感的场景。”

## 七、二面核心表达

### 1 分钟短版

“上次面试后，我发现这个岗位不是单纯看会不会 AI 工具，而是看能不能把 AI 落到真实销售流程里。所以我做了一个外贸客户邮件线索自动录入与跟进建议系统。它可以把客户邮件抽取成结构化线索，写入数据库，在页面展示，并保留人工审核和 AI 判断日志。这个 demo 覆盖了前端、后端、数据库、AI 抽取、审核和日志，后续可以接邮箱 Webhook、N8N、飞书多维表格或正式 CRM。”

### 3 分钟完整版

“这个项目来自上次面试里提到的外贸 CRM 场景。外贸销售经常会收到客户邮件，邮件里可能包含姓名、公司、国家、产品需求、预算、交期、联系方式等信息。如果完全人工录入，效率低，而且后续跟进优先级不清晰。

所以我做了一个最小闭环 demo：销售把客户邮件粘贴进系统，后端调用 AI 抽取结构化字段，包括客户、公司、国家、需求、预算、紧急程度和跟进建议。系统不会让 AI 直接替销售发消息，而是先把结果写入待审核状态。销售可以在详情页查看原始邮件、抽取字段、判断依据和置信度，然后人工确认、修改或拒绝。

技术上，我用 FastAPI 做后端接口，SQLite 做数据存储，前端做邮件录入、线索列表和详情审核页面。数据库里除了客户线索表，还单独保存 AI 抽取日志和审核记录，方便之后复盘 AI 为什么判定这个客户是高意向。

我这样设计主要是考虑外贸客户关系比较敏感，AI 误判或者自动发送都可能影响客户信任。所以这版 demo 把 AI 定位成销售辅助，而不是替代人工。

后续如果进入真实业务，可以把输入端接邮箱 Webhook 或 N8N，把输出端接飞书多维表格或正式 CRM，再扩展 GEO 社媒获客线索，让社媒评论、私信、官网询盘都进入同一套线索跟进流程。”

---

## 八、项目边界提醒

这个 4 天计划的目标是“可闭环、可展示、可解释”，不是上线级产品。

如果中途时间不够，优先级永远是：

主闭环 > 日志和审核 > README 和讲稿 > Docker > CSV > N8N/飞书/GEO 扩展说明。

不要为了显得高级，把项目做成每块都半成品。面试里，一个能跑通的小系统，比五个只讲概念的高级模块更有说服力。
